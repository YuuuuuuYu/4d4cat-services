# TechBlog Expert Skill Guide

이 서비스는 외부 기술 블로그 피드의 안정적인 수집 관리, 무한 스크롤 최적화 및 동시성을 고려한 통계 데이터 관리를 목표로 합니다. 관련 작업을 수행할 때 다음 지침을 엄격히 준수하십시오.

## 🧠 핵심 처리 지침 (Core Instructions)

### 1. 스케줄링 및 데이터 라이프사이클 관리
- **수집 스케줄러**: 매일 08:00, 13:00, 22:00 (KST)에 동작되도록 크론식을 설정하십시오. `@Scheduled(cron = "0 0 8,13,22 * * *")`
- **미노출 전환**: 피드 수집 시 최신 피드 목록에 없는 기존 DB 포스트들은 사용자에게 보여지지 않도록 `is_exposed = false`로 소프트 처리하십시오.
- **삭제 스케줄러**: 매일 00:00 (KST)에 `is_exposed = false`인 포스트를 일괄 삭제(DELETE) 하십시오.

### 2. 가상 스레드 기반 병렬 피드 수집
- 피드 데이터 수집 로직 구현 시 반드시 **Java 21 가상 스레드(Virtual Threads)**를 활용하십시오. 블로킹 I/O 대기 시간을 최소화해야 합니다.

### 3. 고유 식별자 및 통계 테이블 맵핑
- **회사 식별자**: 영문명 기반 **Slug**를 PK 또는 Unique 속성으로 활용하십시오.
- **포스트 식별자**: Auto Increment 방식의 ID를 PK로 사용하십시오.
- **클릭 통계**: 조회수는 별도 `TechBlogPostStat` 테이블에 저장되며, 조회수 갱신 API 호출 시 `UPDATE table SET click_count = click_count + 1 WHERE post_id = ?` 형태의 원자적 쿼리를 우선 고려하여 동시성 문제를 방지하십시오.

### 4. 커서 기반 무한 스크롤 (Cursor-based Pagination)
- 조회 API는 데이터가 밀리거나 중복 노출되는 문제를 막기 위해 **커서(Cursor)** 기반으로 페이징을 구현하십시오.
- 첫 로딩 시 반환되는 데이터의 사이즈(Limit)는 기본 **5개**로 제한하십시오.

## 🔍 작업 전 체크리스트
- [ ] 수집 스케줄러(8, 13, 22시)와 삭제 스케줄러(0시)가 타임존(KST) 기반으로 명확하게 분리 및 설정되었는가?
- [ ] 파싱된 피드에 없는 포스트를 미노출 처리(false)하는 상태 갱신 로직이 존재하는가?
- [ ] 회사 조회 조건은 URL Slug를 기반으로 동작하는가?
- [ ] 통계 테이블(TechBlogPostStat)의 조회수 증가 로직이 다중 요청 상황에서 안전하게(원자적으로) 구현되었는가?
- [ ] 피드 목록 조회 API가 오프셋 기반이 아닌 커서 기반으로 작동하며 기본 크기가 5인가?

## 🧪 테스트 전략 (Testing Strategy)
- **라이프사이클 통합 테스트**: 미노출 상태 전환 및 0시 물리적 삭제 로직이 의도대로 동작하는지 검증합니다.
- **통계 동시성 테스트**: 여러 스레드에서 동시에 단일 포스트 클릭수(`click_count`) 증가 요청을 보낼 때 카운트가 유실되지 않고 정확히 누적되는지 테스트합니다.
- **무한 스크롤 페이징 테스트**: 특정 커서 값을 기준으로 그 이전/이후의 5개 데이터가 정확하게 조회되는지 테스트합니다.
